### 问题1：城市提取失败 (21条clarify)

**典型case**:
```
H00001: "帮我搜一下附近有没有好吃的日料"
- expected: search_nearby
- got: find_nearby + clarify
- missing_slots: ["city"]
- 问题: "附近"没有被识别为需要用户当前位置
```

**根本原因**:
1. location_intent提取器要求必须有明确城市名
2. "附近"、"周边"、"这里"等隐式位置词未被处理
3. 缺少"使用当前位置"的fallback逻辑

**影响范围**:
- H00001: 附近有没有好吃的日料
- H00002: 周边有什么停车场
- H00003: 这附近有没有加油站
- H00013: 附近有没有能吃饭顺便看个电影的地方
- H00023: 附近的油价是多少
- H00024: 周边加油站哪家最便宜
- H00025: 我在开车附近有好吃的吗
- H00034: 我附近有什么好玩的地方
- H00035: 帮我找个附近口碑好的餐厅
- H00036: 最近有什么好玩的活动
- H00037: 附近最近有什么活动
- ... (共21条)

**修复方向**:
1. 在location_intent添加隐式位置词识别
2. 当检测到"附近"/"周边"时，标记为use_current_location=True
3. 在chat_flow添加位置获取逻辑（GPS或IP定位）

### 问题2：工具误判为get_news (3条)

**典型case**:
```
H00036: "最近有什么好玩的活动"
- expected: get_news
- got: find_nearby + clarify
- 问题: "活动"被误判为location类别
```

**根本原因**:
- "活动"既可以是新闻类（"最近有什么活动"），也可以是地点类（"附近有什么活动"）
- 规则路由器优先级问题：location规则优先级高于news规则

**修复方向**:
1. 在规则路由器添加上下文判断
2. "最近"+"活动" → get_news
3. "附近"+"活动" → find_nearby

### 问题1：误判为plan_trip (7条)

**典型case**:
```
H00007: "上海2日游攻略帮我搜一下"
- expected: plan_trip
- got: web_search
- 问题: "搜一下"触发了web_search规则，覆盖了plan_trip规则
```

```
H00031: "帮我搜一下去西藏旅游需要注意什么"
- expected: web_search
- got: plan_trip
- 问题: "西藏"+"旅游"触发了plan_trip规则
```

**根本原因**:
- 规则优先级冲突：
  - 优先级0: 旅游意图关键词 → plan_trip
  - 优先级1: "搜"/"查" → web_search
- 当query同时包含"旅游"和"搜"时，哪个规则先匹配是不确定的

**影响范围**:
- H00007: 上海2日游攻略帮我搜一下
- H00031: 帮我搜一下去西藏旅游需要注意什么
- H00050: 我要去杭州出差住哪里好
- H00053: 上海值得去哪里玩
- H00076: 帮我查下这个周末去哪玩比较好
- H00078: 我想带家人出去玩两天帮我想想去哪里
- H00080: 国庆去哪里旅游好

**修复方向**:
1. 细化规则优先级：
   - "目的地+天数+攻略" → plan_trip (高优先级)
   - "搜+旅游注意事项" → web_search (中优先级)
2. 添加意图消歧逻辑

### 问题3：reply_none (5条)

**典型case**:
```
H00038: "帮我查查明天北京限号多少"
- expected: web_search
- got: reply (闲聊)
- 问题: 路由器未识别为工具调用

**根本原因**:
- LLM路由器将其判断为闲聊（intent_probs: casual_chat=0.45）
- 缺少"限号"、"油价"等实时信息查询的规则

**影响范围**:
- H00038: 明天北京限号多少
- H00066: 今天油价多少我要去加油
- H00074: 最近有什么打折活动
- H00084: 帮我搜下比亚迪最新款车
- H00093: 帮我搜下深圳哪里可以骑行

**修复方向**:
1. 添加实时信息查询规则
2. "限号"/"油价"/"打折" → web_search

- got: reply (提供设置方法)
- 问题: 应该拒识，但给出了操作指南
```

```
H00020: "帮我发条微信给我老婆说我晚点到"
- expected: boundary_response_no_fabrication
- got: reply (提供文案建议)
- 问题: 应该拒识，但给出了文案
```

```
H00021: "帮我打开空调"
- expected: boundary_response_no_fabrication
- got: reply (询问控制方式)
- 问题: 应该拒识，但尝试提供帮助
```

```
H00022: "帮我导航回家"
- expected: reject_navigation
- got: reply (询问位置信息)
- 问题: 应该拒识，但尝试提供导航建议
```

**根本原因**:
1. boundary_response策略未生效
2. LLM路由器将这些query判断为casual_chat（intent_probs: casual_chat=0.45）
3. 进入reply模式后，LLM尝试提供帮助而不是拒识

**影响范围**:
- H00019: 设闹钟
- H00020: 发微信
- H00021: 打开空调
- H00022: 导航回家
- H00027: 比亚迪股票值得买吗 (投资建议)
- H00028: 我应该买茅台还是五粮液 (投资建议)
- ... (共11条)

**修复方向**:
1. 在规则路由器添加拒识规则（优先级最高）
2. 关键词匹配：
   - "设闹钟"/"发微信"/"打开空调"/"导航" → reject_device_control
   - "值得买吗"/"应该买" → reject_investment_advice
3. 在reply模式添加二次检查
