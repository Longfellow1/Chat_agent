# Streaming 流式输出需求文档

**需求类型**: 功能增强  
**优先级**: P1  
**预估工作量**: 0.5 天  
**文档日期**: 2026-03-05  

---

## 背景与目标

当前对话链路为同步阻塞模式，用户发起请求后需等待工具调用和 LLM 生成全部完成，才能一次性看到回复，等待体验差。

LM Studio 本身已支持 OpenAI 兼容的 Streaming 接口，但框架层未使用该能力。

**目标**：在不改动任何现有逻辑的前提下，新增流式输出路径，让用户在 LLM 生成过程中实时看到内容。

---

## 设计原则

1. **只新增，不修改**：所有改动为纯新增，现有 `/chat` 接口和相关逻辑完全不动
2. **只流式化最后一步**：Planner 的路由和参数提取 LLM 调用保持同步，只将最终输出给用户的 Post-Processing LLM 调用改为流式
3. **并行接口**：新增 `/chat/stream` 接口与原 `/chat` 并存，互不影响

## 实现要求

### 1. lm_studio_client.py

新增 `generate_stream` 方法，与现有 `generate` 方法并存：

```python
def generate_stream(self, messages: list, **kwargs):
    """
    流式生成，返回 generator
    逐块 yield str 类型的文本片段
    """
    response = self._client.chat.completions.create(
        model=self.model,
        messages=messages,
        stream=True,
        **kwargs
    )
    for chunk in response:
        content = chunk.choices[0].delta.content
        if content:
            yield content
```

**注意**：
- 复用现有的 `self._client` 和 `self.model`，不引入新的连接
- 超时控制沿用现有配置，不单独设置

### 2. chat_flow.py

新增 `run_stream` 异步方法，与现有 `run` 方法并存：

```python
async def run_stream(self, query: str, session_id: str):
    """
    流式对话，前置流程复用同步逻辑，只将最终 LLM 输出改为流式
    yield str 类型的文本片段
    """
    # 1. 前置流程：查询重写、前置规则、路由、工具调用
    #    全部复用现有同步逻辑（不改动）
    pipeline_result = await self._run_pre_pipeline(query, session_id)

    # 2. 如果前置流程已有最终回复（如拒绝、澄清等），直接 yield 返回
    if pipeline_result.final_text:
        yield pipeline_result.final_text
        return

    # 3. 只有最终输出给用户的 LLM 调用改为流式
    messages = self._build_post_messages(pipeline_result)
    async for chunk in self.llm_client.generate_stream(messages):
        yield chunk

    # 4. 流式结束后，执行会话持久化（不影响用户侧）
    await self._persist_session(session_id, pipeline_result)
```

**注意**：
- `_run_pre_pipeline` 应封装现有 `run()` 方法中工具调用完成前的所有逻辑
- 如果当前 `run()` 方法不易拆分，可以先完整跑一次 `run()` 拿到工具结果，再单独对 Post-Processing 步骤调用 `generate_stream`，优先保证正确性

### 3. server.py

新增 `/chat/stream` 路由，使用 SSE（Server-Sent Events）格式：

```python
from fastapi.responses import StreamingResponse

@app.post("/chat/stream")
async def chat_stream(request: ChatRequest):
    """
    流式对话接口
    返回格式：SSE (text/event-stream)
    每个数据块格式：data: {文本片段}\n\n
    结束标志：data: [DONE]\n\n
    """
    async def generate():
        try:
            async for chunk in chat_flow.run_stream(
                query=request.query,
                session_id=request.session_id
            ):
                yield f"data: {chunk}\n\n"
        except Exception as e:
            yield f"data: [ERROR] {str(e)}\n\n"
        finally:
            yield "data: [DONE]\n\n"

    return StreamingResponse(
        generate(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",  # 禁用 Nginx 缓冲，确保实时推送
        }
    )
```

**注意**：
- 请求体复用现有 `ChatRequest` schema，不新增字段
- 错误时通过 `[ERROR]` 标志通知客户端，不直接抛异常中断 SSE 连接

## 验收标准

| 验收项 | 标准 |
|--------|------|
| 功能正确性 | `/chat/stream` 能正常返回流式内容，内容与 `/chat` 一致 |
| 非破坏性 | 原有 `/chat` 接口行为完全不变，所有现有测试通过 |
| 流式效果 | 用户侧能看到文字逐步出现，而非等待后一次性显示 |
| 错误处理 | LLM 调用失败时，SSE 连接正常关闭，返回 `[ERROR]` 标志 |
| 工具调用路径 | 流式接口的工具调用结果与同步接口完全一致 |
