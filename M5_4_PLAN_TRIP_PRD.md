## 一、背景与问题

### 1.1 当前状态

**现有实现** (Legacy):
- **数据源**: Tavily 网页搜索
- **实现方式**: 搜索 "{destination} 2日游 攻略 景点 交通 住宿 美食"
- **输出**: 简单的 D1/D2 行程建议 + 来源链接
- **问题**:
  - ❌ 缺少地图数据支持（距离、路线、交通方式）
  - ❌ 缺少实时POI信息（营业时间、评分、联系方式）
  - ❌ 行程规划不够智能（未考虑地理位置优化）
  - ❌ 内容质量依赖搜索结果，不稳定

**参考项目**: Mcp_test (`/Users/Harland/Documents/evaluation/Mcp_test`)
- **架构**: Langchain Agent + 高德地图MCP + 小红书MCP
- **优势**: 
  - ✅ 地图数据支持（POI搜索、路线规划、导航）
  - ✅ 社交内容支持（旅游攻略、美食推荐、热门打卡地）
  - ✅ Agent自主决策工具调用
  - ✅ 结构化行程输出

### 1.2 用户痛点

**车展场景**:
1. 用户: "帮我规划一个上海2日游"
2. 期望: 
   - 每天的景点安排（考虑地理位置，减少路程）
   - 景点之间的交通方式和时间
   - 推荐的餐厅和美食
   - 住宿建议
3. 当前输出: "D1 上海核心景点；D2 上海城市漫游"（过于简单）

**核心问题**:
- 行程规划与地图信息强相关，但当前实现未使用地图API
- 缺少结构化的行程数据（时间、地点、路线）
- 无法提供实时的POI信息（营业时间、评分）

## 二、产品目标

### 2.1 核心目标

**P0 (Must Have - 车展前必须完成)**:
1. 集成高德地图MCP，提供POI搜索和基础地图数据
2. 实现基于地理位置聚类的行程规划算法
3. 输出"时段+顺序"格式的行程（上午/下午/晚上，不精确到分钟）
4. 支持流式输出（先输出第一天，边生成边播报）
5. 延迟控制在10秒以内（含流式首字延迟TTFT）

**P1 (Should Have - 车展后优化)**:
1. 集成百度地图MCP作为降级方案
2. 支持3-5日行程规划（当前仅2日）
3. 支持用户偏好参数（美食、文化、自然风光等）
4. 优化景点选择算法（避免回头路、考虑营业时间）

**P2 (Nice to Have - Q2规划)**:
1. 按需路线规划（用户追问时调用 maps_route_planning）
2. 行程导出功能（PDF、图片、日历）
3. 个性化推荐（基于历史偏好）

### 2.2 成功指标

**定量指标**:
- 行程规划准确率: ≥ 85%
- 地图数据使用率: ≥ 80%
- 流式首字延迟(TTFT): ≤ 2s
- 完整行程生成延迟: ≤ 10s
- 用户满意度: ≥ 4.0/5.0

**定性指标**:
- 行程安排合理（地理位置优化，景点间交通时间 ≤ 45分钟）
- 信息完整（景点、餐厅、交通方式）
- 内容自然（LLM重写后流畅易读）

## 三、产品方案

### 3.1 核心流程

```
用户查询: "帮我规划一个上海2日游"
  ↓
[意图识别] plan_trip
  ↓
[参数提取] destination="上海", days=2
  ↓
[工具调用] plan_trip(destination="上海", days=2)
  ↓
[行程规划引擎]
  ├─ Step 1: 搜索目的地热门景点 (高德MCP text_search)
  ├─ Step 2: 地理位置聚类（按区域分组景点）
  ├─ Step 3: 规划每日行程（上午/下午/晚上时段分配）
  ├─ Step 4: 推荐餐厅 (高德MCP around_search，按需调用)
  └─ Step 5: 生成结构化行程（流式输出）
  ↓
[Content Rewriter] LLM重写为自然语言（流式）
  ↓
返回用户（边生成边播报）
```

**关键决策**:
- ❌ 不默认调用 `maps_route_planning`（API额度限制，仅用户追问时调用）
- ✅ 使用距离估算 + 文字描述交通方式
- ✅ 流式输出（先第一天，再第二天）

### 3.2 输出格式（修正版）

**核心原则**: 用"节奏感"替代"时刻表"

用户要的是可执行的行程骨架，不是精确时刻表。精确时间计算只在用户主动追问时触发。

**推荐格式** (时段+顺序):

```
上海2日游行程规划：

第1天 - 经典地标游

上午
• [第一站] 外滩（中山东一路）
  建议游览2小时，欣赏黄浦江两岸风光

• [午餐] 南翔馒头店（豫园路85号）
  上海本帮菜，评分4.5分

下午
• [第二站] 东方明珠（世纪大道1号）
  交通：地铁约30分钟
  建议游览2小时

晚上
• [晚餐] 新天地周边餐厅
  推荐：上海菜、西餐

第2天 - 文化艺术游
...
```

**避免的格式** (精确时刻表):
```
❌ 09:00-11:00 外滩
❌ 11:30-13:00 午餐
❌ 13:30 出发前往东方明珠
```

**原因**: LLM需要连续做时间加法，一旦算错，整个时间链断掉，用户反而觉得规划不可信。

**按需精确计算**: 
- 用户追问: "我10点到外滩，几点能去东方明珠"
- 此时才调用 `maps_route_planning` 做导航级计算

### 3.3 车载场景约束（新增）

**延迟要求**:
- 流式首字延迟(TTFT): ≤ 2s
- 完整行程生成: ≤ 10s
- 必须支持流式输出（先输出第一天框架，边生成边播报）

**输出适配**:
- 确认车机是否有屏幕显示
- 如果纯语音播报，输出格式需要进一步简化：
  - 去掉地址、评分等细节
  - 只保留景点名和交通方式
  - 示例: "上午去外滩，中午在豫园吃饭，下午坐地铁去东方明珠"

### 3.4 降级策略

**Provider Chain**: 高德MCP → 百度地图MCP → Tavily → 明确提示

| 场景 | 主方案 | 降级方案 | 兜底方案 |
|------|--------|---------|---------|
| POI搜索 | 高德MCP | 百度地图MCP | Tavily搜索 |
| 周边推荐 | 高德MCP | 百度地图MCP | Tavily搜索 |
| 路线规划 | 高德MCP | 百度地图MCP | 文字描述 |
| 全部失败 | - | - | 明确提示用户 |

**关键修正**:
- ❌ 移除Mock数据（仅用于开发测试）
- ✅ 全部provider失败时返回明确提示，引导用户换一种方式查询
- ✅ 示例: "抱歉，暂时无法获取{destination}的行程信息，您可以尝试搜索'{destination} 旅游攻略'"

## 四、技术要求

### 4.1 API集成

**高德地图MCP** (主要):
- `maps_text_search`: 搜索景点（必须）
- `maps_around_search`: 周边搜索（按需）
- `maps_route_planning`: 路线规划（仅用户追问时调用）

**百度地图MCP** (降级):
- `place_search`: 搜索景点
- `place_detail`: 获取详情
- `direction`: 路线规划

**Tavily** (兜底):
- 搜索旅游攻略和推荐

### 4.2 性能要求

- **流式首字延迟(TTFT)**: ≤ 2s
- **完整行程生成**: ≤ 10s
- **API调用**: ≤ 8次/请求（不含路线规划）
- **缓存**: 热门目的地行程缓存24小时
- **并发**: 支持多个API并行调用

### 4.3 数据质量

- **景点数量**: 每天3-5个景点
- **餐厅推荐**: 每天2-3个餐厅（可选）
- **交通时间准确性**: 景点间交通时间 ≤ 45分钟
- **信息完整性**: POI信息完整率 ≥ 90%

## 十、出行方式策略设计（新增）

### 10.1 信号识别（Parser层）

**自驾信号词**:
```python
SELF_DRIVE_SIGNALS = [
    "自驾", "自驾游", "开车", "驾车", "自己开",
    "自驾出发", "从家开车", "带车去"
]
```

**识别逻辑**:
- 命中自驾信号词 → `travel_mode = "driving"`
- 未命中 → `travel_mode = "transit"`（默认公共交通）

**参数透传**:
```python
plan_trip(destination="上海", days=2, travel_mode="driving")
```

### 10.2 出行方式对行程规划的影响

两种模式在三个维度上有实质差异：

| 维度 | 自驾模式 | 公共交通模式 |
|------|---------|-------------|
| 景点聚类 | 可跨区域，按路线串联 | 优先同地铁线/步行圈聚类 |
| 交通描述 | 驾车X分钟/X公里 | 地铁X号线/公交/步行 |
| 路线规划API | `maps_route_planning(mode=driving)` | `maps_route_planning(mode=transit)`，按需调用 |

### 10.3 JSON Schema更新

在每个stop里增加 `transit_to_next` 的结构化字段，区分两种模式的输出：

**自驾模式示例**:
```json
{
  "travel_mode": "driving",
  "itinerary": [
    {
      "day": 1,
      "theme": "经典地标游",
      "sessions": [
        {
          "period": "上午",
          "stops": [
            {
              "order": 1,
              "type": "景点",
              "name": "外滩",
              "address": "中山东一路",
              "duration_minutes": 120,
              "transit_to_next": {
                "mode": "driving",
                "duration_minutes": 15,
                "distance_km": 3.2,
                "description": "驾车约15分钟，3.2公里"
              }
            }
          ]
        }
      ]
    }
  ]
}
```

**公共交通模式示例**:
```json
{
  "transit_to_next": {
    "mode": "transit",
    "duration_minutes": 30,
    "description": "地铁10号线转2号线，约30分钟"
  }
}
```

### 10.4 自驾模式的特殊处理

**1. 停车信息（已砍掉）**

**决策**: 停车信息优先级不高，Phase 1 不提供。

**原因**: 
- 用户要的重点是规划信息，而并非什么都跟他说
- 信息过多，用户记不住
- API调用成本高

**2. 路线连贯性**

自驾模式下景点顺序要考虑驾车路线不走回头路。

**实现方式**:
1. 获取所有景点的经纬度
2. 对于每对相邻景点，调用 `maps_route_planning(mode=driving, origin=A, destination=B)`
3. 获取驾车时间和距离
4. LLM根据结果续写交通描述

**示例**:
```python
# 景点A: 外滩 (121.490, 31.240)
# 景点B: 东方明珠 (121.506, 31.245)

result = amap_mcp.route_planning(
    mode="driving",
    origin="121.490,31.240",
    destination="121.506,31.245"
)

# 返回: {"duration": 15, "distance": 3200}
# 续写: "驾车约15分钟，3.2公里"
```
