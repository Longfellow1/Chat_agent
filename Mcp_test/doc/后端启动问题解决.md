# 🔧 后端启动问题解决

## 问题诊断

**错误：** `Failed to fetch`

**原因：** 前端无法连接到后端服务器（后端未运行或启动失败）

---

## ✅ 解决方案：分别启动前后端

### 步骤 1: 启动后端（终端 1）

打开第一个终端，运行：

```bash
cd /Users/Harland/Documents/Mcp_test
npm run server
```

**预期输出：**
```
🚀 启动服务器，初始化 Agent...
🚀 初始化行程规划 Agent...
📦 加载 MCP 工具...
✅ 高德地图 MCP 客户端已连接（或跳过）
✅ 行程规划 Agent 初始化完成
🚀 行程规划 API 服务器运行在 http://localhost:3001
📝 健康检查: http://localhost:3001/health
```

**如果看到错误：**
- 检查 `.env` 文件中的 `OPENAI_API_KEY` 是否正确配置
- 查看具体的错误信息

### 步骤 2: 启动前端（终端 2）

打开第二个终端，运行：

```bash
cd /Users/Harland/Documents/Mcp_test
npm run client
```

**预期输出：**
```
VITE v7.x.x  ready in xxx ms

➜  Local:   http://localhost:5173/
➜  Network: use --host to expose
```

### 步骤 3: 验证连接

1. **检查后端健康状态**
   ```bash
   curl http://localhost:3001/health
   ```
   应该返回：`{"status":"ok","message":"Trip Planner API is running"}`

2. **访问前端**
   - 打开浏览器访问：http://localhost:5173
   - 现在应该可以正常使用了

## 🔍 常见错误排查

### 错误 1: Agent 初始化失败

**症状：**
```
❌ Agent 初始化失败，服务器仍将启动但功能可能受限
```

**原因：** `OPENAI_API_KEY` 未配置或无效

**解决：**
1. 检查 `.env` 文件是否存在
2. 确认 `OPENAI_API_KEY` 已正确填入
3. 确认 API Key 格式正确（以 `sk-` 开头）

### 错误 2: 端口被占用

**症状：**
```
Error: listen EADDRINUSE: address already in use :::3001
```

**解决：**
```bash
# 查找占用端口的进程
lsof -ti:3001

# 杀死进程（替换 PID 为实际进程 ID）
kill -9 PID

# 或者修改 .env 中的端口
PORT=3002
```

### 错误 3: 模块找不到

**症状：**
```
Error: Cannot find module 'xxx'
```

**解决：**
```bash
# 重新安装依赖
npm install
```

### 错误 4: 前端代理配置问题

**检查：** `client/vite.config.js` 中的代理配置是否正确

## 📋 推荐的启动方式

### 方式一：两个终端分别启动（推荐，便于查看日志）

**终端 1 - 后端：**
```bash
cd /Users/Harland/Documents/Mcp_test
npm run server
```

**终端 2 - 前端：**
```bash
cd /Users/Harland/Documents/Mcp_test
npm run client
```

**优点：**
- 可以分别查看前后端的日志
- 错误信息更清晰
- 便于调试

### 方式二：使用 concurrently（一个命令同时启动）

```bash
npm run dev
```

**注意：** 如果使用这种方式，确保两个服务都能正常启动。如果有错误，建议用方式一分别启动。

---

## ✅ 验证步骤

启动后，按以下顺序验证：

1. **后端健康检查**
   ```bash
   curl http://localhost:3001/health
   ```

2. **查看后端日志**
   - 应该看到 Agent 初始化成功的消息
   - 如果有 MCP 工具，应该看到工具列表

3. **访问前端**
   - http://localhost:5173
   - 应该看到行程规划界面

4. **测试 API 调用**
   - 在前端输入测试用例
   - 提交后查看后端日志中的请求处理情况

## 💡 提示

如果 `npm run dev` 不工作，**建议使用分别启动的方式**，这样可以：
- 更清楚地看到每个服务的日志
- 更容易定位问题
- 可以单独重启某个服务
