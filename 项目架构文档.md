# Agent Service 项目架构文档

## 一、项目概述

Agent Service 是一个智能对话代理服务，支持多种实时工具调用（天气、股票、新闻、地点搜索、旅游规划、网页搜索等），具备意图识别、工具路由、多Provider降级、会话管理等能力。

### 核心特性
- **多工具支持**: 天气、股票、新闻、地点搜索、旅游规划、网页搜索
- **智能路由**: 基于规则和LLM的混合路由策略
- **Provider链**: 自动降级机制（主API → 备用API → Mock）
- **会话管理**: 支持上下文重写和指代消解
- **可观测性**: 完整的Trace日志系统
- **多后端支持**: LM Studio、Coze等

### 技术栈
- **语言**: Python 3.10+
- **Web框架**: FastAPI + Uvicorn
- **LLM后端**: LM Studio / Coze
- **外部API**: 和风天气、新浪财经、高德地图、Tavily搜索、百度搜索等

## 三、核心模块详解

### 3.1 应用层 (app/)

#### 3.1.1 API服务器 (server.py)
- FastAPI应用入口
- 健康检查接口 `/health`
- 对话接口 `/chat`
- 全局中间件（Trace ID、错误处理）
- 自动端口查找机制

#### 3.1.2 对话流程编排 (chat_flow.py)
核心流程：
1. **查询重写**: 基于会话上下文的指代消解
2. **前置规则检查**: 危机检测、违规检测、无意义输入过滤
3. **意图路由**: 规则路由 + LLM路由（可配置）
4. **工具调用**: 参数提取 → 工具执行 → 结果后处理
5. **后置规则检查**: 反幻觉保护
6. **会话持久化**: 保存上下文信息

关键特性：
- 支持规则路由和LLM路由混合策略
- 工具结果可选LLM后处理（提升自然度）
- 超时控制（分工具类型配置）
- 降级策略（工具失败 → LLM回复）


### 3.2 领域层 (domain/)

#### 3.2.1 意图识别与路由 (intents/)

**router.py - 基于规则的工具路由器**
- 支持6种工具：get_weather、get_news、get_stock、find_nearby、plan_trip、web_search
- 评分机制：
  - 强关键词 +3分
  - 弱关键词 +1分
  - 正则模式 +4分
- 冲突解决：
  - 天气指数 vs 股票指数消歧
  - 地点搜索 vs 信息查询消歧
  - 垂直工具优先于通用搜索

**web_search_router.py - Web搜索路由器**
- 防止误触发机制（排除模式）
- 强/弱关键词组合判断
- 垂直工具优先策略

**encyclopedia_router.py - 百科路由器**
- 区分百科查询 vs 实时搜索
- 支持概念解释、人物介绍等

#### 3.2.2 地点搜索领域 (location/)

**LocationIntent模型**
```python
LocationIntent:
  - city: 城市
  - location: 具体地点（街道/地标）
  - category: POI类别
  - keyword: 搜索关键词
  - confidence: 置信度
```

**核心组件**:
- **parser.py**: 地点意图解析器（规则+模式匹配）
- **capability.py**: POI能力定义（结构化 vs 非结构化）
- **result_processor.py**: 结果处理链（排序、过滤、去重）
- **templates.py**: 响应模板生成器
- **dictionaries.py**: 地点词典（城市、地标、POI类别）

**处理流程**:
1. 解析用户查询 → LocationIntent
2. 判断POI类别能力（结构化/非结构化）
3. 调用对应数据源（高德MCP/Web搜索）
4. 结果处理（排序、过滤、模板化）

#### 3.2.3 工具规划与执行 (tools/)

**planner.py - 工具参数规划器**
- 基于规则的参数提取
- 参数归一化（城市别名、股票代码映射）
- 必填槽位检查

**planner_v2.py - 增强版规划器**
- 集成LocationIntent支持
- 自动补全缺失参数
- 置信度评估

**executor.py - 工具执行器**
- 统一调用MCPToolGateway
- 异常处理和降级

**query_preprocessor.py - 查询预处理器**
- 关键词提取
- 噪声过滤
- 实体识别

   - 备: Tavily搜索

2. **get_stock (股票行情)**
   - 主: 新浪财经API
   - 备: Tavily搜索
   - 兜底: Mock

3. **web_search (网页搜索)**
   - 主: Tavily API
   - 备: 百度搜索MCP
   - 兜底: Mock

4. **get_news (新闻资讯)**
   - 主: Tavily News API
   - 兜底: Mock

5. **find_nearby (地点搜索)**
   - 主: 高德地图MCP
   - 备: Tavily搜索
   - 兜底: Mock

6. **plan_trip (旅游规划)**
   - 主: Tavily搜索
   - 兜底: Mock

**Provider链管理器 (provider_chain.py)**
- 自动降级机制
- 熔断器集成
- 性能指标收集
- 动态配置更新

**核心特性**:
- 超时降级（可配置）
- 错误码匹配降级
- 降级链追踪
- Provider可用性检查

#### 3.3.3 高德地图MCP客户端 (amap_mcp_client.py)

**功能**:
- POI搜索（关键词+城市/坐标）
- 地理编码（地址→坐标）
- 逆地理编码（坐标→地址）
- 结果排序和过滤

**集成LocationIntent**:
- 自动参数转换
- 结果重排序
- 模板化响应

**session_store.py - 会话存储**
- 内存存储（带TTL）
- 支持上下文信息持久化
- 用于查询重写和指代消解

**tracing.py - Trace日志**
- 每次请求生成唯一trace_id
- JSONL格式日志
- 包含完整的请求链路信息

**circuit_breaker.py - 熔断器**
- 失败率阈值触发
- 半开状态恢复
- 防止雪崩效应

**result_cache.py - 结果缓存**
- 减少重复API调用
- TTL过期机制
- 内存缓存实现

**quota_monitor.py - 配额监控**
- API调用次数统计
- 配额预警
- 限流保护

## 四、数据流图

### 4.1 完整请求流程

```
用户请求
  ↓
[FastAPI Server] 生成trace_id
  ↓
[ChatFlow.run()]
  ↓
[1. 查询重写] rewrite_query() → effective_query
  ↓
[2. 前置规则检查]
  ├─ detect_end_chat() → 结束对话
  ├─ detect_meaningless_noise() → 拒绝
  ├─ detect_illegal_sensitive() → 拒绝
  └─ detect_crisis() → 危机干预
  ↓
[3. 意图路由]
  ├─ 规则路由: route_query()
  └─ LLM路由: _route_with_llm() (可选)
  ↓
[4. 工具调用路径]
  ├─ [4.1 参数提取]
  │   ├─ 规则提取: extract_rule_tool_args()
  │   ├─ LLM提取: _extract_slots_with_llm()
  │   └─ LocationIntent: parse_location_intent() (find_nearby)
  │
  ├─ [4.2 参数检查]
  │   └─ 缺失必填参数 → 返回clarify响应
  │
  ├─ [4.3 工具执行]
  │   └─ ToolExecutor.execute()
  │       └─ MCPToolGateway.invoke()
  │           ├─ 主Provider调用
  │           ├─ 失败 → 备用Provider
  │           └─ 全失败 → Mock兜底
  │
  └─ [4.4 结果后处理]
      └─ LLM后处理 (可选): 格式化、自然语言生成
  ↓
[5. LLM回复路径] (工具路由失败时)
  └─ LLMClient.generate()
  ↓
[6. 后置规则检查]
  └─ should_block_free_realtime_reply() → 反幻觉保护
  ↓
[7. 会话持久化]
  └─ session_store.upsert()
  ↓
[8. 返回响应] ChatResponse
  └─ 包含: final_text, tool_name, latency_ms, trace_id等
```

### 4.2 Provider链降级流程

```
工具调用请求
  ↓
[ProviderChainManager.execute()]
  ↓
遍历Provider链 (按优先级)
  ↓
[Provider 1] 主Provider
  ├─ 可用性检查 → 不可用 → 记录fallback → 下一个
  ├─ 执行调用
  │   ├─ 成功 → 返回结果 (附带fallback_chain)
  │   ├─ 超时 → 检查fallback_on_timeout
  │   │   ├─ true → 记录fallback → 下一个
  │   │   └─ false → 返回错误
  │   └─ 错误 → 检查fallback_on_error
  │       ├─ true → 记录fallback → 下一个
  │       └─ false → 返回错误
  ↓
[Provider 2] 备用Provider
  └─ (同上流程)
  ↓
[Provider N] Mock Provider
  └─ 兜底返回
  ↓
返回最终结果
```

## 五、关键设计决策

### 5.1 混合路由策略

**问题**: 纯规则路由覆盖不全，纯LLM路由延迟高且不稳定

**解决方案**:
- 默认使用规则路由（低延迟、高准确率）
- 可选LLM路由增强（处理复杂query）
- 规则安全覆盖（防止LLM误判）

**配置**:
```bash
USE_LLM_ROUTER=true          # 启用LLM路由
FAST_ROUTE_HINT=true         # 规则快速路径优化
USE_ROUTE_OVERRIDE=true      # 规则安全覆盖
```

### 5.2 Provider链设计

**问题**: 单一API不稳定，需要降级机制

**解决方案**:
- Provider抽象基类（统一接口）
- 链式管理器（自动降级）
- 熔断器保护（防止雪崩）
- 指标收集（监控和优化）

**优势**:
- 高可用性（多级降级）
- 可扩展性（新增Provider无需改动业务代码）
- 可观测性（完整的降级链追踪）

### 5.3 LocationIntent设计

**问题**: 地点搜索参数复杂，规则难以覆盖

**解决方案**:
- 独立的LocationIntent模型
- 规则+模式匹配的解析器
- 结果处理链（排序、过滤、模板化）
- POI能力路由（结构化 vs 非结构化）

**优势**:
- 高准确率（专门优化）
- 低延迟（无需LLM）
- 可扩展（易于添加新规则）

### 5.4 工具结果后处理

**问题**: 工具原始输出不够自然，用户体验差

**解决方案**:
- 可选的LLM后处理（TOOL_POST_LLM=true）
- 分工具类型的超时配置
- 智能跳过机制（已足够简洁的结果）

**配置**:
```bash
TOOL_POST_LLM=true                    # 启用后处理
TOOL_POST_LLM_TIMEOUT_SEC=18          # 默认超时
TOOL_POST_LLM_TIMEOUT_STOCK_SEC=6     # 股票超时
TOOL_POST_LLM_TIMEOUT_WEATHER_SEC=6   # 天气超时
TOOL_POST_LLM_TIMEOUT_SEARCH_SEC=5    # 搜索超时
```

## 七、部署与运行

### 7.1 本地开发

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env 文件

# 3. 启动LM Studio (如使用本地模型)
# 加载模型: qwen2.5-7b-instruct

# 4. 启动服务
./scripts/run_agent_server.sh

# 5. 测试
curl http://127.0.0.1:8000/health
./scripts/call_chat_api.sh http://127.0.0.1:8000 "北京今天天气怎么样"
```

### 7.2 CLI模式

```bash
# 单次查询
./scripts/run_agent_local.sh "今天天气怎么样"

# 带会话ID的多轮对话
./scripts/call_chat_api.sh http://127.0.0.1:8000 "郑州天气怎么样" session-1
./scripts/call_chat_api.sh http://127.0.0.1:8000 "明天呢" session-1
```

### 7.3 生产部署

```bash
# 使用Docker (推荐)
docker build -t agent-service .
docker run -p 8000:8000 --env-file .env agent-service

# 或使用systemd
sudo cp agent-service.service /etc/systemd/system/
sudo systemctl enable agent-service
sudo systemctl start agent-service
```

## 十一、常见问题

### Q1: 如何添加新工具？

1. 在`domain/tools/planner.py`中添加工具定义
2. 在`infra/tool_clients/mcp_gateway.py`中实现工具逻辑
3. 在`domain/intents/router.py`中添加路由规则
4. 添加单元测试和集成测试

### Q2: 如何添加新Provider？

1. 继承`ToolProvider`基类
2. 实现`execute()`方法
3. 在`provider_chain.py`中注册
4. 配置Provider链

### Q3: 如何调试路由问题？

1. 查看Trace日志中的`route_source`和`intent_probs`
2. 使用`get_routing_debug_info()`获取详细评分
3. 调整规则权重或添加新规则

### Q4: 如何优化延迟？

1. 启用`FAST_ROUTE_HINT`快速路径
2. 调整超时配置（降低超时时间）
3. 启用结果缓存
4. 禁用工具后处理（TOOL_POST_LLM=false）

### Q5: 如何处理API配额限制？

1. 配置`quota_monitor.py`监控配额
2. 启用结果缓存减少调用
3. 配置多个Provider降级
4. 实现限流保护

## 十二、Mcp_test 项目架构

### 项目概述

Mcp_test 是一个基于 Langchain 1.0 框架的行程规划助手演示项目，通过 MCP（Model Context Protocol）集成高德地图和小红书等外部服务，展示如何在 LLM 应用中集成多个 MCP 工具。

### 核心特性
- **Langchain 1.0 集成**: 使用最新的 Langchain 框架
- **MCP 工具集成**: 支持高德地图、小红书等 MCP 服务
- **Agent 自动工具选择**: LLM 自动判断使用哪个工具
- **前后端分离**: React 前端 + Node.js 后端
- **多传输方式**: 支持 HTTP/SSE 和 STDIO 两种 MCP 传输方式

### 核心模块详解

#### 1. Langchain Agent (server/agent/planner.js)

**职责**: 协调 LLM 和 MCP 工具，处理用户行程规划请求

**核心流程**:
```
用户查询 (如：想去北京玩3天)
    ↓
Agent 接收查询
    ↓
LLM 分析意图 (旅游规划)
    ↓
自动选择工具
    ├─ 高德地图: 搜索景点、规划路线
    └─ 小红书: 搜索攻略、推荐内容
    ↓
整合工具结果
    ↓
生成行程规划
    ↓
返回用户
```

**关键特性**:
- 自动工具选择（无需手动指定）
- 多轮工具调用（可连续调用多个工具）
- 结果整合和优化

#### 2. MCP 客户端 (server/mcp/)

**高德地图 MCP** (amap.js)
- 地点搜索（POI 搜索）
- 路线规划（驾车、步行、公交）
- 地理编码（地址→坐标）
- 天气查询

**小红书 MCP** (xhs.js)
- 攻略搜索
- 内容推荐
- 用户评价

**MCP 传输方式**:
```
方式1: HTTP/SSE (远程服务器)
  ├─ URL: https://mcp.service.com/v1
  ├─ Headers: Authorization, API-Key
  └─ 适用: 云端 MCP 服务

方式2: STDIO (本地服务器)
  ├─ Command: npx -y @amap/mcp-server
  ├─ Args: [--port, 3000]
  └─ 适用: 本地开发、私有部署
```

#### 3. Express 服务器 (server/index.js)

**API 端点**:
- `GET /health` - 健康检查
- `POST /api/plan` - 行程规划接口

**请求/响应示例**:

## 十三、Chatflow 项目架构

### 项目概述

Chatflow 是一个基于字节跳动 Coze 平台的多意图对话工作流项目，支持无目标闲聊、常识/百科、情绪表达、实时数据检索等多种对话场景。项目采用可视化工作流编排方式，通过节点组合实现复杂的对话逻辑。

### 核心特性
- **多意图识别**: 自动识别用户意图（闲聊、百科、实时查询等）
- **工作流编排**: 可视化节点编排，支持条件分支和循环
- **长期记忆**: 支持用户信息持久化和上下文记忆
- **实时数据集成**: 集成天气、新闻、股票等实时数据源
- **安全防护**: 内置违法内容检测和人身安全风险识别

### 项目结构

```
Chatflow-Chat_2_0224-draft-1273/
└── Chatflow-Chat_2_0224-draft-1273/
    ├── MANIFEST.yml                    # 项目元数据
    └── workflow/
        └── Chat_2_0224-draft.yaml      # 工作流定义 (4765 行)
```

### 工作流架构

#### 工作流节点类型

| 节点类型 | 功能 | 说明 |
|---------|------|------|
| start | 开始 | 工作流入口，定义输入参数 |
| end | 结束 | 工作流出口，返回结果 |
| llm | LLM 调用 | 调用大语言模型 |
| intent | 意图识别 | 识别用户意图 |
| condition | 条件分支 | 根据条件选择分支 |
| plugin | 插件调用 | 调用外部 API（天气、新闻等） |
| code | 代码执行 | 执行 Python 代码 |
| ltm_write | 长期记忆写入 | 保存用户信息 |
| output | 输出 | 返回消息给用户 |
| comment | 注释 | 文档说明 |

#### 核心工作流

```
开始 (USER_INPUT)
  ↓
获取当前时间 (code 节点)
  ↓
意图识别 (intent 节点)
  ├─ 意图1: 查询天气
  ├─ 意图2: 新闻资讯
  ├─ 意图3: 网络搜索
  ├─ 意图4: 股票查询
  ├─ 意图5: 行程规划
  ├─ 意图6: 周边搜索
  └─ 意图7: 其他
  ↓
条件分支 (condition 节点)
  ├─ 分支1: 天气查询 → DayWeather 插件 → 天气总结 LLM
  ├─ 分支2: 新闻查询 → 新闻 LLM
  ├─ 分支3: 网络搜索 → 搜索 LLM
  ├─ 分支4: 股票查询 → 股票 LLM
  ├─ 分支5: 行程规划 → 行程 LLM
  ├─ 分支6: 周边搜索 → 周边 LLM
  ├─ 分支7: 常识/百科 → 百科 LLM
  └─ 分支8: 无目标闲聊 → 闲聊 LLM
  ↓
长期记忆写入 (ltm_write 节点)
  ↓
输出结果 (output 节点)
  ↓
结束
```

### 关键节点详解

#### 1. 意图识别节点 (intent 节点)

**功能**: 使用 LLM 识别用户意图

**意图定义**:
```yaml
意图1: 查询天气
  - 触发条件: 包含城市 + 天气关键词
  - 示例: "北京今天天气怎么样"

意图2: 新闻资讯查询
  - 触发条件: 新闻、资讯、时事、热点
  - 示例: "最新的科技新闻"

意图3: 网络搜索
  - 触发条件: 信息查询、最新信息、可变信息
  - 示例: "iPhone 15 价格多少"

意图4: 股票查询
  - 触发条件: 股票实时股价、涨跌情况
  - 示例: "查一下茅台股价"

意图5: 行程规划
  - 触发条件: 出行路线、旅游行程、游玩攻略
  - 示例: "帮我规划北京3天旅游"

意图6: 周边搜索
  - 触发条件: 具体地点附近的美食、景点等
  - 示例: "五道口附近有什么好吃的"

意图7: 其他
  - 触发条件: 闲聊、常识、百科等
  - 示例: "你好"、"什么是黑洞"
```

#### 2. LLM 节点配置

**常识/百科查询 LLM**:
```yaml
模型: 豆包·1.6·自动深度思考
温度: 0.3 (精确模式)
系统提示词: |
  你是一个擅长聊常识和百科的轻松朋友
  - 回答生活常识、日常技巧
  - 讲解科学概念、文化常识
  - 用口语化、简洁的方式表达
  - 最多加1个轻语气词（哦、呀、啦）
```

**无目标闲聊 LLM**:
```yaml
模型: 豆包·1.5·Pro·32k
温度: 0.8 (创意模式)
系统提示词: |
  你是一个亲切自然的无目标闲聊伙伴
  - 主动发起轻松话题
  - 自然延续日常闲聊
  - 保持无焦点、开放式互动
  - 调用长期记忆时用"我记得..."开头
```

#### 3. 长期记忆 (LTM) 节点

**功能**: 保存和调用用户信息

**保存内容**:
- 用户名字
- 用户爱好
- 用户计划
- 用户身份信息
- 对话历史摘要

**调用方式**:
```
在闲聊 LLM 中引用: {{outputList}}
格式: [{"date": "2026-02-01", "output": "用户信息"}]
```

#### 4. 条件分支节点

**逻辑**:
```
if intent_id == 1:
  → 天气查询分支
elif intent_id == 2:
  → 新闻查询分支
elif intent_id == 3:
  → 网络搜索分支
...
else:
  → 默认分支
```

```

  ↓
┌─────────────────────────────────────┐
│  获取当前时间 (Code 节点)            │

```

### 关于 LangChain 1.0 迁移

#### 结论: 不建议现阶段整体迁移

**核心理由**:

当前架构的核心竞争力在于**规则路由 + LLM 兜底**的混合策略，这已经解决了 LangChain Agent 最大的痛点——工具选择不可控。

**性能对比**:

| 方案 | 链路 | 延迟 | 准确率 |
|------|------|------|--------|
| LangChain Agent | 用户 → LLM 分析意图 → 自动选工具 → 工具返回 → LLM 整合 | ~2-3s | 70-80% |
| 当前方案 | 用户 → 规则路由 (< 10ms) → 参数提取 → 工具执行 → LLM 后处理 | ~1-1.5s | 85-90% |

**关键差异**:
- 规则路由省掉一次 LLM 调用（节省 500-800ms）
- 车展场景对延迟敏感，这个差异很明显
- 规则路由准确率更高（针对性优化）

#### LangChain 1.0 能力评估

| LangChain 能力 | 是否需要 | 建议 |
|---------------|---------|------|
| LCEL 链式调用语法 | 可选 | 局部引入即可，不必整体迁移 |
| 更成熟的 Memory 管理 | 暂不需要 | 当前 session_store 够用 |
| 更多内置 Tool 集成 | 不需要 | 自研 Provider 体系更可控 |
| Streaming 支持 | **有价值** | **可以单独引入** |

#### 推荐方案: 选择性引入

**短期（1-2个月）**:
```python
# 仅引入 Streaming 支持
from langchain.callbacks.streaming_stdout import StreamingStdOutCallbackHandler

class ChatFlow:
    def run_streaming(self, req: ChatRequest):
        """流式输出版本"""
        callback = StreamingStdOutCallbackHandler()
        # 保持现有路由逻辑

        # 仅在 LLM 调用时使用 streaming

            return self.llm_client.generate_stream(
                query=req.query,
                callbacks=[callback]
            )
```

**中期（3-6个月）**:

**不建议**:
- ❌ 整体迁移到 LangChain Agent 模式
- ❌ 替换现有的路由体系
- ❌ 使用 LangChain 的工具选择机制
