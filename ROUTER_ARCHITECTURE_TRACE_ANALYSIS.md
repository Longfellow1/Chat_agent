# Router Architecture Trace Analysis

## 当前架构特征

### 架构概览

```
用户 query
    ↓
规则层（RuleBasedRouter）
    ├─ 规则0：旅游意图关键词 → plan_trip（优先级最高）
    ├─ 规则1：目的地 + 时间 → plan_trip
    ├─ 规则2：位置 + 类别 → find_nearby
    ├─ 规则3：天气关键词 + 位置 → get_weather
    └─ 未命中 ↓
4B LLM（qwen/qwen3-4b-2507）
    ├─ 意图识别 + 参数提取
    ├─ 置信度评分（0.4/0.5/0.65/0.85）
    └─ 输出：tool + params + confidence
        ↓
置信度检查
    ├─ ≥ 0.7 → 执行
    └─ < 0.7 → 澄清
```

### 核心特征

| 特征 | 实现 | 性能 |
|------|------|------|
| 规则层 | 关键词匹配 | 0ms，100% 准确 |
| LLM 层 | 4B 模型 | 350-400ms，87-90% 准确 |
| 置信度 | 启发式评分 | 0.4-0.9 五档 |
| 兜底 | Parse 失败 → web_search | 不崩溃 |

### 规则层覆盖

| 工具 | 规则数量 | 关键词示例 |
|------|---------|-----------|
| plan_trip | 3 条规则 | "旅游"、"旅行"、"出去玩"、"避暑"等 10 个 |
| find_nearby | 1 条规则 | 位置 + 类别（餐厅、酒店、景点等） |
| get_weather | 1 条规则 | "天气"、"温度"、"下雨"、"晴天"、"风" |
| web_search | 无规则 | 全部走 LLM |
| get_news | 无规则 | 全部走 LLM |
| get_stock | 无规则 | 全部走 LLM |

### 性能指标

| 指标 | 规则层 | LLM 层 | 整体 |
|------|--------|--------|------|
| 延迟 | 0ms | 350-400ms | 305ms（平均）|
| 准确率 | 100% | 87-90% | 87-90% |
| 覆盖率 | ~25% | ~75% | 100% |

## 初步观察

### 1. 规则层覆盖率差异巨大

| 工具 | 规则命中 | LLM 处理 | 规则覆盖率 |
|------|---------|---------|-----------|
| plan_trip | 7/10 | 3/10 | 70% |
| find_nearby | 5/10 | 5/10 | 50% |
| get_weather | 5/10 | 5/10 | 50% |
| web_search | 1/10 | 9/10 | 10% |
| get_news | 0/10 | 10/10 | 0% |
| get_stock | 0/10 | 10/10 | 0% |

**问题**：
- get_news 和 get_stock 完全没有规则覆盖
- web_search 规则覆盖率极低（10%）
- 这三个工具全部依赖 LLM，延迟高且不稳定

### 2. 延迟分布不均

| 工具 | 平均延迟 | 规则延迟 | LLM 延迟 |
|------|---------|---------|---------|
| plan_trip | 183.7ms | 0ms | ~612ms |
| find_nearby | 200.3ms | 0ms | ~401ms |
| get_weather | 177.5ms | 0ms | ~355ms |
| web_search | 319.5ms | 0ms | ~355ms |
| get_news | 354.4ms | - | ~354ms |
| get_stock | 350.3ms | - | ~350ms |

**问题**：
- get_news 和 get_stock 每次都要 350ms+
- 即使是简单的 "新闻"、"股票" 单字查询也要走 LLM
- 没有快速路径

### 3. LLM 低置信度问题严重

**低置信度 case（source=llm_low_confidence）**：
- find_nearby: 4/5 LLM case 低置信度
- get_weather: 2/5 LLM case 低置信度
- get_news: 3/10 低置信度
- get_stock: 8/10 低置信度（80%！）

**问题**：
- get_stock 的 80% case 低置信度，说明 4B 对股票意图识别很弱
- 低置信度会触发澄清，影响用户体验

### 4. 规则误判

**意外的规则命中**：
- "周末想出去走走" → 规则命中（应该走 LLM）
- "我想去旅游，但不知道去哪" → 规则命中（应该走 LLM）
- "北京有什么好玩的" → 规则命中（应该是 web_search）

**问题**：
- 规则层过于激进，匹配了不该匹配的 case
- "出去走走" 被加入旅游关键词后，导致误判

## 架构问题分析

### 问题 1：规则层不平衡

**现状**：
- plan_trip/find_nearby/get_weather 有规则
- get_news/get_stock/web_search 几乎没有规则

**影响**：
- 后三个工具延迟高（350ms+）
- 用户体验不一致

**建议**：
- 补充 get_news 规则（"新闻"、"最新消息"、"今天发生了什么"）
- 补充 get_stock 规则（"股票"、"股价"、"市场行情"）
- 这些高频词不应该走 LLM

### 问题 2：规则关键词过于宽泛

**现状**：
- "出去走走" 被加入旅游关键词
- 导致 "周末想出去走走" 误判

**影响**：
- 规则层误判，覆盖了 LLM 的正确判断
- 规则优先级高于 LLM，无法纠正

**建议**：
- 规则关键词应该更保守
- "出去走走" 这种口语化表达应该走 LLM
- 只保留明确的关键词（"旅游"、"旅行"、"度假"）

### 问题 3：LLM 对某些工具识别弱

**现状**：
- get_stock: 80% 低置信度
- get_news: 30% 低置信度

**影响**：
- 大量澄清，用户体验差
- 4B 模型对金融/新闻领域识别能力弱

**建议**：
- 这两个工具应该加强规则层
- 不要依赖 4B 推理

### 问题 4：规则层和 LLM 层职责不清

**现状**：
- 规则层：处理明确的关键词匹配
- LLM 层：处理复杂语义和隐含意图

**问题**：
- 规则层关键词过多，侵入 LLM 领域
- LLM 层处理简单查询（"新闻"、"股票"），浪费资源

**建议**：
- 规则层：只处理高频、明确的 query
- LLM 层：处理复杂、模糊的 query
- 中间层：简单但不在规则中的 query（需要新策略）

## 架构优化建议

### 方案 1：三层架构（规则 + 简单 LLM + 复杂 LLM）

```
用户 query
    ↓
规则层（高频明确 query）
    ├─ 命中 → 返回（0ms）
    └─ 未命中 ↓
简单分类器（单字/高频词）
    ├─ 命中 → 返回（50ms，轻量模型）
    └─ 未命中 ↓
4B LLM（复杂语义）
    ├─ 置信度 ≥ 0.7 → 返回（350ms）
    └─ 置信度 < 0.7 → 澄清
```

**优点**：
- 简单查询（"新闻"、"股票"）走简单分类器（50ms）
- 复杂查询走 4B LLM（350ms）
- 延迟分层，用户体验更好

**缺点**：
- 需要训练简单分类器
- 架构复杂度增加

### 方案 2：扩展规则层（保守关键词 + 高频词）

```
用户 query
    ↓
规则层（扩展）
    ├─ 保守关键词（"旅游"、"天气"）→ 返回
    ├─ 高频单字（"新闻"、"股票"）→ 返回
    └─ 未命中 ↓
4B LLM（复杂语义）
    ├─ 置信度 ≥ 0.7 → 返回
    └─ 置信度 < 0.7 → 澄清
```

**优点**：
- 架构简单，只需扩展规则
- 立竿见影，快速部署

**缺点**：
- 规则维护成本高
- 无法处理变体（"最新新闻"、"股市行情"）

### 方案 3：规则 + 4B + 7B 升级

```
用户 query
    ↓
规则层（保守关键词）
    ├─ 命中 → 返回（0ms）
    └─ 未命中 ↓
4B LLM（快速意图识别）
    ├─ 置信度 ≥ 0.8 → 返回（350ms）
    ├─ 0.5 ≤ 置信度 < 0.8 → 升级到 7B
    └─ 置信度 < 0.5 → 澄清
        ↓
7B LLM（复杂语义）
    ├─ 置信度 ≥ 0.7 → 返回（900ms）
    └─ 置信度 < 0.7 → 澄清
```

**优点**：
- 简单 query 走 4B（350ms）
- 复杂 query 走 7B（900ms）
- 准确率和延迟平衡

**缺点**：
- 需要部署两个模型
- 成本增加

## 推荐方案

**短期（车展前）**：方案 2 - 扩展规则层

**理由**：
- 快速部署，无需训练新模型
- 覆盖高频 query（"新闻"、"股票"）
- 立即降低延迟

**具体动作**：
1. 补充 get_news 规则（10 个高频词）
2. 补充 get_stock 规则（10 个高频词）
3. 收紧 plan_trip 规则（移除 "出去走走" 等口语化词）
4. 监控规则命中率和误判率

**长期（车展后）**：方案 3 - 4B + 7B 升级

**理由**：
- 准确率和延迟最优
- 适合生产环境长期运行
- 可以根据置信度动态选择模型
