# 失败Case分类分析

## 目的
将22条失败case按照"是否在第一层设计范围内"分类，为真实流量数据分析做准备。

## 分类标准

### 第一层范围内（可通过规则优化解决）
- **覆盖问题**：地标/商圈在高德存在，但未加入白名单
- **同义词问题**：用户词汇与字典不匹配，需补充映射

### 第一层边界（需要架构升级）
- **歧义问题**：多城市同名地标，单纯规则无法消歧
- **多anchor问题**：query包含多个地点，需要选择逻辑
- **复杂结构问题**：query结构超出当前pattern覆盖范围

### 超出第一层（需要LLM或第二层）
- **隐含需求**：需要语义理解才能推断category
- **上下文依赖**：需要多轮对话历史

### 类别C：第一层边界问题（需要架构升级）- 1条

这些case超出了当前规则系统的能力边界，需要更复杂的逻辑。

#### C1. 多anchor选择问题（1条）
```
11. "深圳市南山区科技园地铁站出口的快餐店"
    - 失败原因：提取了"地铁站出口"，而不是"科技园"
    - 根本原因：query包含多个POI（科技园、地铁站、出口），当前逻辑选择最后一个
    - 第一层能力：只能提取单个anchor
    - 解决方案：
      1. 短期：优先选择更早/更长的anchor（"科技园" > "地铁站出口"）
      2. 中期：支持多anchor提取 + 排序逻辑
      3. 长期：LLM理解用户真实意图（可能是"科技园地铁站"）
    - 优先级：低（边界case，占比<2%）
```

**C类总结**：
- 数量：1/20 (5%)
- 特征：需要复杂的选择/排序逻辑
- 解决成本：中（需要架构调整）
- 预期提升：+1条 → 70/80 (87.5%)

### 类别D：复杂结构问题（第一层边界）- 8条

这些case的query结构超出了当前pattern的覆盖范围。

#### D1. 短地名无后缀（8条）
```
12-19. 各种"XX的YY"结构，其中XX是短地名（2-3字）无明显POI后缀
    - 示例：天津滨江道、深圳东门、青岛台东、广州天河
    - 失败原因：这些地名没有"路/街/广场"等后缀，不匹配POI pattern
    - 根本原因：当前逻辑依赖POI后缀识别，对于纯地名无能为力
    - 第一层能力：只能识别带后缀的POI
    - 解决方案：
      1. 短期：加白名单（见类别A）
      2. 中期：建立"常用商圈/区域"白名单
      3. 长期：LLM理解地名语义
    - 优先级：高（占比40%，但可通过白名单解决）
```

**D类总结**：
- 数量：8/20 (40%)
- 特征：地名无明显后缀，依赖白名单
- 解决成本：低（加白名单）→ 实际归入类别A
- 预期提升：已计入A类
